# 企业级 RAG SaaS 平台

## 0. 销售版本说明

当前母版项目支持两种对外版本：

- **企业版**：单后台版本，适合个人、小团队、单企业私有部署
- **服务商版**：双后台版本，适合服务商、多客户托管和 SaaS 运营

### 企业版（单后台）

特点：

- 一个后台入口
- 包含账户管理
- 包含知识库、Python 脚本、模型配置、企业定制化、日志、问答测试
- 前台聊天页正常保留
- 不开放平台总后台路由

默认入口：

- 后台：`/tenant`
- 登录：`/login`
- 聊天：`/chat`

### 服务商版（双后台）

特点：

- 平台总后台 + 租户后台
- 平台总后台：企业账户管理、平台总日志
- 租户后台：企业自己的知识库、脚本、模型、日志、问答测试
- 前台聊天页正常保留

默认入口：

- 平台后台：`/admin`
- 租户后台：`/tenant`
- 登录：`/login`
- 聊天：`/chat`

### 一键导出

母版项目内置导出接口，用于生成可交付版本包：

- `GET /api/admin/release-profiles`
- `POST /api/admin/release/export`
- `GET /api/admin/release/download?zip_name=...`

导出逻辑当前采用**轻量版本化**：

- 共用一套代码底座
- 根据版本写入不同的 `edition / deployment_mode`
- 根据版本切换页面入口和平台路由
- 自动生成版本说明文档
- 打包为 zip 文件供你直接下载和售卖

## 1. 项目定位

这是一个面向企业知识场景的多租户 RAG SaaS 平台，不是单一业务问答壳。

平台提供三类能力：

- 平台总后台：管理企业账户、查看平台总日志
- 租户后台：管理企业自己的知识库、采集脚本、模型配置、品牌配置、日志与问答测试
- 前台聊天页：终端用户登录后直接使用企业自己的知识助手

适用场景示例：

- 企业知识库问答
- SOP / 制度 / 产品文档检索
- 私有资料问答助手
- 多企业独立配置的 AI 助手平台

## 3. 技术架构

### 3.1 后端框架

- FastAPI：API 服务与页面路由
- LangGraph：问答主链路编排
- SQLite：业务数据存储（当前版本）
- Redis：语义缓存后端（已接入，未配置时自动回退内存缓存）

### 3.2 RAG 检索链

- Qdrant：向量数据库
- Embedding：支持云端与本地 fallback
- Rerank：支持云端与本地 fallback
- Guardrails：输入输出护栏
- Observability：请求日志、护栏事件、聊天日志、运行状态统计

### 3.3 文档处理能力

当前版本已经不再只支持 Markdown，知识库入口支持多种企业常见文档：

- Markdown `.md`
- 纯文本 `.txt`
- HTML `.html/.htm`
- PDF `.pdf`
- Word `.docx`
- PowerPoint `.pptx`
- Excel `.xlsx/.xls`
- 图片 `.png/.jpg/.jpeg`

文档处理链路包含：

- LangChain 文档加载器工厂
- 标题分块
- 章节分块
- 语义分块
- PDF 表格抽取
- 图片 OCR
- 扫描版 PDF OCR 兜底
- 跨页短内容自动合并

当前策略重点不是“把文件读出来”而已，而是尽量保留企业文档的结构信息：

- 标题层级尽量保留
- 章节上下文尽量保留
- 长段内容会按语义句群切块
- 短页内容会做跨页合并
- 表格尽量整体抽取
- 扫描版 PDF 会尝试 OCR

这使得企业资料在入库后更适合做检索与问答，而不是被粗暴打散成碎片。

### 3.4 LangChain 组件层

当前版本已经把 LangChain 从“零散引用”升级成了独立组件层，主要承担三件事：

- `DocumentLoader`：统一接入 PDF / Word / PPT / HTML / Markdown / Excel
- `TextSplitter`：统一承接标题分块和递归分块
- `Retriever Adapter`：把当前项目的检索结果导出成 LangChain Retriever

这一层的价值不是替代 LangGraph，而是把以下能力标准化：

- 文档 ingest
- 分块策略
- 检索组件对接

当前架构关系是：

- `LangGraph`：负责编排、状态流、检索纠错、模型调用
- `LangChain`：负责文档加载、切块、检索组件适配
- `Qdrant / BM25 / Rerank`：负责底层召回与重排

### 3.5 分块策略

当前版本已经把分块策略升级成“企业文档友好”模式：

- 标题分块：优先按 H1 / H2 / H3 / H4 切开大章节
- 章节分块：保留章节标题路径，避免段落脱离上下文
- 语义分块：长段内容按句群和关键词重叠做轻量语义切分
- 表格整体保留：Markdown / PDF 表格尽量作为完整知识块处理
- 跨页内容修复：对分页导致的短碎片做自动合并

分块不是越碎越好。当前策略的目标是：

- 不把企业制度、SOP、公告切得太碎
- 不让跨页文档在分页处断裂
- 不把表格切散成无法理解的零碎片段
- 在保留结构的同时，让检索粒度足够细

### 3.7 检索与知识调度

系统当前的检索链不是单一向量检索，而是：

- 向量检索
- BM25 稀疏检索
- 混合召回（Hybrid Retrieval）
- Rerank 重排序
- Query Rewrite
- 检索质量评估
- 失败自动重试

当前混合检索不是简单把两路结果拼起来，而是做了进一步增强：

- 根据问题类型自动识别检索画像
- 动态调整稠密检索 / 稀疏检索权重
- 对编号、术语、流程、FAQ 等不同类型问题使用不同策略
- 结合分数归一化和 RRF 思路做更稳的融合打分

当前内置的问题类型包括：

- `identifier_lookup`：编号、单号、接口名、参数名、字段名等精确定位型问题
- `keyword_exact`：关键词强依赖型问题
- `process_policy`：制度、流程、审批、规范类问题
- `faq_semantic`：解释、总结、说明、介绍类问题

默认情况下：

- 精确定位型问题会提高 BM25 权重
- FAQ / 解释型问题会提高语义向量权重
- 流程制度类问题会使用相对平衡的混合策略

在此基础上，系统还内置了更强的检索纠错链路：

- Query Rewrite：会根据问题画像做关键词扩展和轻量改写
- 检索路径路由：不同问题会优先走不同检索后端
- 检索质量评估：根据命中条数、最高分、平均分判断结果是否可靠
- 分级重试：第一次结果差时，不只是扩大召回条数，而是会切换检索策略

当前默认的分级重试策略为：

- 第 1 次：按问题画像选择默认检索路径
- 第 2 次：保持混合检索，但扩大召回并做更宽松改写
- 第 3 次：切到更偏 BM25 的精确召回路径，适合编号、制度、字段、术语类问题

三层知识库会同时参与检索：

- `L3 热库`：默认优先级最高，适合命中最新变化
- `L2 增量库`：补充阶段性与项目性知识
- `L1 基础库`：提供长期稳定依据和兜底知识

系统会结合：

- 层级权重
- 语义召回
- BM25 召回
- 重排序结果

综合决定最终喂给模型的知识片段。

## 4. 多租户隔离设计

系统按 `tenant_id` 隔离企业数据。

### 4.1 配置隔离

每个租户拥有自己的配置目录：

- `data/tenants/<tenant_id>/`

其中包含：

- `app_config.json`
- `system_prompt.md`
- `model_config.json`
- `api_keys.txt`
- `retrieval_config.json`
- `crawler_config.json`

### 4.2 知识库隔离

每个租户拥有自己的知识目录：

- `knowledge/<tenant_id>/`

知识库固定三层：

- `L3/`：L3 热库
- `L2/`：L2 增量库
- `L1/`：L1 基础库

### 4.3 向量检索隔离

每个租户拥有自己的 Qdrant collection。

默认命名规则：

- `tenant_<tenant_id>_knowledge`

### 4.4 日志隔离

以下日志均按 `tenant_id` 进行隔离：

- 请求日志
- 聊天日志
- 护栏事件
- 采集执行历史

## 5. Python 脚本系统

平台内置通用采集执行器与轻量调度器，给不懂代码的运营人员也能直接使用。

支持能力：

- 配置数据源
- 选择来源类型（网站 / API / 数据库）
- 选择目标知识库层级
- 配置抓取规则
- 手动执行
- 自动调度
- 查看执行历史
- 查看调度器状态

### 5.1 后台每个字段怎么填

在“Python脚本设置”里，一条采集源通常要填写这些字段：

- 来源名称：给自己看的名字，例如“公司公告页”“产品文档 API”
- 来源 ID：系统内部唯一标识，建议只用英文小写、数字和下划线，例如 `notice_center`
- 来源地址：网页地址、API 地址或数据库连接目标
- 来源类型：
  - 网站：抓网页正文
  - API：抓接口返回内容
  - 数据库：抓数据库查询结果
- 目标知识库：
  - L3 热库：公告、快讯、临时更新
  - L2 增量库：项目资料、周期任务、专题内容
  - L1 基础库：制度、SOP、标准资料
- 抓取频率：每小时、每天、每周
- 自动执行：打开后调度器会按频率自动跑
- 人工审核：打开后先进入审核队列，不会直接入库
- 规则：告诉系统“正文在哪、哪些内容要保留、哪些内容要过滤”

### 5.2 最常用的规则怎么写

当前规则是按“每行一条”的形式写，支持 4 个核心参数：

- `selector=`：从网页里抓哪些元素
- `include=`：只保留包含这些关键词的内容
- `exclude=`：过滤掉包含这些关键词的内容
- `limit=`：最多保留多少行

最常见的规则写法：

```text
selector=.article h2,.article p
include=制度|流程|审批
exclude=广告|推荐阅读
limit=20
```

你可以直接这样理解：

- `selector=.article h2,.article p`
  - 去网页里找 `.article` 这个区域下的标题和正文段落
- `include=制度|流程|审批`
  - 只保留包含“制度”“流程”“审批”这些词的内容
- `exclude=广告|推荐阅读`
  - 看到“广告”“推荐阅读”就删掉
- `limit=20`
  - 最多保留 20 行，避免整页都塞进知识库

### 5.3 上传文档时系统会自动做什么

当你上传 PDF / Word / PPT / HTML / Excel / 图片等文件后，系统会自动执行：

1. 识别文件类型
2. 用对应解析器加载正文
3. 尽量保留标题与结构信息
4. 按章节、段落和长度切块
5. 对扫描版 PDF 或图片尝试 OCR
6. 对 PDF 表格做抽取
7. 对跨页短内容做自动合并
8. 再进入三层知识库与向量检索链

所以企业管理员不需要先把所有资料手工转成 Markdown，很多常见文档可以直接上传。

### 5.4 不懂代码时，直接照这几个模板填

#### 模板 1：抓新闻 / 公告页正文

适合：公告、公司新闻、通知中心

```text
selector=article h1,article h2,article p,.article h1,.article h2,.article p
exclude=广告|相关推荐|上一篇|下一篇|扫码|免责声明
limit=30
```

#### 模板 2：抓制度 / 流程 / SOP 页面

适合：制度文档、操作流程、财务审批说明

```text
selector=article h1,article h2,article h3,article p,.article h1,.article h2,.article h3,.article p
include=制度|流程|SOP|步骤|审批|提交|规范
exclude=广告|推荐阅读|友情链接
limit=40
```

#### 模板 3：抓帮助中心 / FAQ

适合：帮助中心、常见问题、产品文档

```text
selector=.content h1,.content h2,.content h3,.content p,.content li
exclude=广告|相关推荐|联系客服|返回顶部
limit=50
```

### 5.4 规则写错了会怎样

系统不会因为规则写错就把整个后台弄崩，但可能出现下面几种结果：

- 抓不到正文：通常是 `selector` 没写对
- 抓到一堆废话：通常是没写 `exclude`
- 正文太少：通常是 `include` 条件太严
- 正文太多：通常是没写 `limit`

最简单的排查方法：

1. 先只写 `selector`
2. 手动执行一次
3. 看执行历史和入库结果
4. 再慢慢补 `exclude`
5. 最后再补 `include` 和 `limit`

### 5.5 给不懂技术的人一句建议

如果你完全不会写规则，就按这个顺序来：

1. 先选“网站”
2. 先填来源地址
3. 先用“公告页模板”或“SOP 模板”
4. 先手动执行一次
5. 结果正常后再打开“自动执行”

这样最稳，不容易一上来就抓错数据。
